## WHY: 特殊入力を追加しても普段の変換と翌日の候補を壊さない

日付・時刻・計算結果を単に候補へ足すだけでは、通常の`きょう → 今日`が消えたり、確定した`2026/07/21`が学習されて翌日も古い日付として先頭へ残ったりする。特殊候補を最初の候補ページへ置きつつ、既存の学習・ユーザー辞書・モデル・システム辞書を残し、入力時に再計算すべき値だけを学習対象から外す必要があった。

## HOW: 共有Rustエンジンで生成・優先順位・学習可否を一度だけ決める

- `SpecialConversionRewriter`が日時、4桁日時、元号、計算、Unicode、バージョン、顔文字、読み訂正を生成する。実行時はOSのローカル時刻、単体試験は固定日時を同じ計算処理へ渡す。
- 候補順を「学習 → ユーザー辞書 → 特殊変換 → モデル → システム辞書 → 通常表記変換」とし、通常モデルは特殊候補の有無にかかわらず実行する。
- `CandidateSource::SpecialConversion`だけ公開候補の読みを持たせず、Enter、数字選択、分節移動、変換範囲変更を含む確定経路で学習させない。通常開始と候補再構築は同じ`candidate_list_from_annotated`を使う。
- macOSとfcitx5へ個別ロジックを増やさず、両方が既に利用する共有エンジンへ統合した。
- 顔文字・読み訂正・日時/元号ロジックはMozc由来で、`THIRD_PARTY_LICENSES`へ出典を追記した。OSローカル時刻には既に`Cargo.lock`に存在した`libc 0.2.182`を直接依存として宣言し、新しいパッケージは追加していない。

検討中に見つかった「特殊候補があるとモデル変換を省略する案」と「特殊候補を通常候補と同じ読みで学習する案」は、既存候補の消失と古い日時の残留を起こすため採用していない。

## WHAT: Space変換または既存の直接入力で使える機能

| 入力 | 確認した候補・動作 |
|---|---|
| `きょう`、`あした`、`きのう`、`あさって` | 西暦、和暦、曜日。日・月・うるう年の境界も固定時計で確認 |
| `げつようび`〜`にちようび` | 次の該当曜日の日付 |
| `ことし`、`きょねん`、`らいねん` | 西暦と和暦の年 |
| `こんげつ`、`せんげつ`、`らいげつ` | 月の数字と`○月`表記 |
| `いま`、`じこく`、`にちじ`、`なう` | 現在時刻または現在日時 |
| `1230` | `12月30日`、`12時30分`、`12時半` |
| `0000`〜`2959` | Mozc準拠の時刻。`3000`以降または分60以上は時刻候補なし |
| `645ねん`、`1989ねん`、`2019ねん` | `大化元年`、切替年の新旧元号候補。2201年以降は元号候補なし |
| `(10+4)*3=`、`100/8=`、`2^10=` | `42`、`12.5`、`1024`。構文不正・ゼロ除算・非有限値は候補なし |
| 数字 | 既存の全角、漢数字、大字、ローマ数字、丸数字、2/8/16進数を維持 |
| `よっと`、`あんどろいど` | 配布辞書から`yacht`、`Android` |
| `あぼがど`、`しゅみれーしょん` | `アボカド`、`シミュレーション` |
| `かおもじ`、`にこにこ`、`しょぼーん` | Mozc由来の読み別顔文字。既存の絵文字・記号変換も維持 |
| `U+611B` | `愛`。サロゲート・範囲外・不正16進数は候補なし |
| `ばーじょん` | `Karukan 0.1.0` |
| `zh/zj/zk/zl`など | 矢印、括弧、約物を既存Zコマンドで直接入力 |

利用方法と条件は`karukan-im/README.md`へ追記した。実装上の対象と役割は`referent-table-google-ime-special-conversion.md`へ先に固定し、最終SHA-256は`8f3c9f7c5470933cb3780337a783f968f40f23e7f87e5badafd863b3ddb55758`である。
